# DISCUSSION

## API definition
API 需要清晰无歧义，这也导致了它缺乏复用性。即不能进行重载，否则语言模型常常会错误调用

## Example
有时不方便用语言表述的样例，通过样例让LLM了解意图是很好的

## Model
对于不同的语言模型，效果差异很明显，GPT4能够有很强的推理能力。对于少量的example就可以推理出类似情况下的解决方案，而GPT3.5在这方面弱很多

## Hyperparameter
尽量使用低想象的参数，这样能确保生成更稳定的代码

## Human feedback
人类的二次提示，能很好的纠正LLM生成的错误代码

## Detail instruction
充满细节的任务描述，如告诉它一些关键的步骤，通常会取得更好的结果，尤其对于能力低的LLM来说。如GPT3.5，因为它难以推断出中间的操作。

## History conversation
对于新的任务，是否需要考虑之前历史任务的对话。有些是需要考虑的，如GPT3.5经常将pose理解为XYZ，忽略了转角。如果上述任务中，人类纠正过了，后续代码就几乎不会犯这样的错误。
但相关的场景描述，其实已经提供了任务相关的信息了。


## Discussion
1. 不同模型reasoning的能力足够，但是对API的理解，形参及返回值类型存在偏差
2. 使用LLM的长程记忆能够有效的理解环境，但是这样会占用大量的token，且错误的回复也会引入额外偏差，即使反馈纠正了
3. 在简单任务上，有时不使用提示词优化的方式，能提供较好的结果，因为对于这些任务，代码量比较少，往往直接生成会有概率猜测正确，而加入thought或者更多的提示词/example会引入额外偏差
4. GPT4的推理能力十分强大，能够生成很可靠且稳定的代码，虽然对于hard的任务，它的初始生成结果不好，但都是固定的错误，如错误理解了API的用法，比较容易经过人类的反馈调整。但是GPT3.5等模型的结果逻辑较为混乱，难以调整。
5. 对于场景的更新，我们测试了两种方式，包括使用原始模型和用一个新的模型及提示词来进行推理，发现使用原有模型的方式更好，这或许是因为原始模型包含更多的上下文信息，scene update的结果也是连贯性的生成的，而使用另外模型的话是query/answer的形式，结果没有使用单个模型准确
